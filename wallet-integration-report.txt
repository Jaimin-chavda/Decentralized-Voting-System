1. INTRODUCTION
────────────────────────────────────────────────────────────────────────────────

   The Decentralized Governance System (DGS) requires users to connect
   a cryptocurrency wallet in order to participate in on-chain governance
   voting. MetaMask — the most widely used browser-extension wallet —
   was chosen as the primary wallet provider.

   This report covers how MetaMask was integrated into the React frontend,
   the authentication flow using message signing, the backend verification
   reference, and the smart-contract interaction layer.


2. TECHNOLOGY STACK USED
────────────────────────────────────────────────────────────────────────────────

   - MetaMask Browser Extension  — Injects the `window.ethereum` provider
   - ethers.js (v6)              — Ethereum library for providers, signers,
                                    and contract interaction
   - React (Hooks)               — Custom hooks for wallet state management
   - Express.js (Node.js)        — Backend reference server for signature
                                    verification
   - Solidity 0.8.27 + Hardhat   — Smart contract & local blockchain

4. HOW METAMASK DETECTION WORKS
────────────────────────────────────────────────────────────────────────────────

   When MetaMask is installed, it injects a global `window.ethereum` object
   into every web page. The application checks for this object on page load.

   Detection Logic (in useWallet.js):
     • On component mount, the hook checks if `window.ethereum` exists and
       if `window.ethereum.isMetaMask` is true.
     • This distinguishes MetaMask from other injected providers (e.g. the
       Hardhat provider which also injects `window.ethereum`).
     • The detection result is stored in the `isMetaMaskInstalled` state
       variable.

   In the UI (WalletConnect.jsx):
     • A 600ms detection delay is applied to give the browser time to inject
       `window.ethereum`. During this time, a spinner with "Detecting
       Wallet..." is shown.
     • If MetaMask is detected, a green banner with "MetaMask Detected" and
       a "Ready" badge is displayed alongside the connect button.
     • If NOT detected, a red banner with "MetaMask Not Found" is shown with
       a link to install MetaMask from metamask.io.


5. WALLET CONNECTION FLOW
────────────────────────────────────────────────────────────────────────────────

   Step 1 — User Clicks "Connect"
     • The user clicks the MetaMask connect button on the WalletConnect page.
     • This calls the `connectWallet()` function from the useWallet hook.

   Step 2 — MetaMask Popup
     • The hook uses `wallet_requestPermissions` (not just `eth_requestAccounts`)
       to ALWAYS force the MetaMask popup, even if the site was previously
       approved. This ensures the user explicitly chooses which account to
       connect each time.
     • Code:
         await window.ethereum.request({
           method: "wallet_requestPermissions",
           params: [{ eth_accounts: {} }],
         });

   Step 3 — Account Retrieved
     • After the user approves, the selected account address is fetched
       using `eth_accounts`.
     • The account address is stored in React state.
     • A `walletConnected` flag is saved to localStorage for auto-reconnect
       on future visits.

   Step 4 — Network Detection
     • The hook fetches the current chain ID using `eth_chainId`.
     • It maps the chain ID to a human-readable network name using a lookup
       table that supports: Ethereum Mainnet, Sepolia Testnet, Polygon,
       Arbitrum, Optimism, BNB Chain, Goerli, and Hardhat Local.
     • The required chain for this app is Sepolia Testnet (chain ID 0xaa36a7).
     • If the user is on a different network, a "Wrong Network" warning is
       displayed with a prompt to switch.

   Step 5 — Connected State UI
     • After connection, the page shows:
         - A green checkmark and "Wallet Connected" header
         - The shortened wallet address (e.g., 0xA1B2...C3D4)
         - Network name and connection status (with a glowing green dot)
         - Wallet type (MetaMask) and full address
     • Action buttons for Sign Message, Read Contract, Go to Homepage,
       and Disconnect are displayed.

   Error Handling:
     • If the user rejects the MetaMask popup (error code 4001), a friendly
       message is shown: "You rejected the connection request."
     • Other errors show: "Failed to connect wallet. Make sure MetaMask is
       unlocked and try again."


6. MESSAGE SIGNING & AUTHENTICATION
────────────────────────────────────────────────────────────────────────────────

   The application uses Ethereum message signing for authentication — a
   standard Web3 practice that proves wallet ownership without any gas cost.

   Frontend Flow (useWallet.js → authenticateWithSignature):
     1. A BrowserProvider is created from `window.ethereum`.
     2. A signer object is obtained from the provider.
     3. A random nonce is generated for replay-attack protection.
     4. A message is constructed:
          "Sign this message to authenticate with GovChain.
           Nonce: <random_number>
           Timestamp: <ISO_timestamp>"
     5. The signer's `signMessage()` method is called, which triggers a
        MetaMask popup asking the user to sign (no gas cost — it is an
        off-chain operation).
     6. The function returns { address, signature, message } which can be
        sent to the backend for verification.

   Backend Verification (server/authVerify.js):
     The reference backend Express server has two endpoints:

     a) POST /api/auth/nonce
        • Receives the wallet address.
        • Generates a cryptographically-random nonce using Node.js
          crypto.randomBytes(16).
        • Stores the nonce in an in-memory Map keyed by wallet address.
        • Returns the full message string for the frontend to sign.

     b) POST /api/auth/verify
        • Receives { address, signature, message } from the frontend.
        • Uses ethers.verifyMessage(message, signature) to recover the
          signer's address from the signature.
        • Compares the recovered address with the claimed address — if they
          match, the user truly owns the wallet.
        • Validates the nonce to prevent replay attacks.
        • Invalidates the nonce after use (one-time use).
        • In production, a JWT or session token would be issued here.


7. AUTO-RECONNECT MECHANISM
────────────────────────────────────────────────────────────────────────────────

   To improve user experience, the wallet auto-reconnects on subsequent
   page visits:

     • When the user connects, a `walletConnected` flag is saved to
       localStorage.
     • On page load, the hook checks if this flag exists.
     • If true AND `window.ethereum.isMetaMask` is true (not Hardhat), it
       silently calls `eth_accounts` (which does NOT open a popup) to
       retrieve previously approved accounts.
     • If MetaMask is locked or the user has revoked access, the localStorage
       flag is cleaned up automatically.

   This design avoids silently picking up Hardhat/localhost accounts when
   running a local dev node.


8. METAMASK EVENT LISTENERS
────────────────────────────────────────────────────────────────────────────────

   The useWallet hook listens for MetaMask events to keep the UI in sync:

   a) accountsChanged
      • Fires when the user switches accounts in MetaMask.
      • If no accounts remain (user locks MetaMask), the wallet is
        auto-disconnected.
      • If a different account is selected, the state updates automatically.

   b) chainChanged
      • Fires when the user switches networks in MetaMask.
      • The chain ID and network name are updated in real-time.
      • If the new network is not Sepolia, the "Wrong Network" warning
        appears immediately.

   Event listeners are properly cleaned up on component unmount to prevent
   memory leaks.


9. SMART CONTRACT INTERACTION VIA WALLET
────────────────────────────────────────────────────────────────────────────────

   The useContract.js hook provides a separate path for interacting with the
   GovChainVoting smart contract on the local Hardhat blockchain:

   Connection:
     • Uses ethers.JsonRpcProvider to connect directly to the local Hardhat
       RPC at http://127.0.0.1:8545.
     • This does NOT require MetaMask — it uses Hardhat's built-in accounts
       directly for easy demo testing.
     • 5 pre-funded Hardhat accounts are loaded (Account 0 = Admin/Deployer,
       Accounts 1-4 = Voters).

   Contract Instance:
     • The contract address and ABI are imported from contractConfig.js.
     • An ethers.Contract instance is created with the selected signer.
     • The contract instance exposes all functions: createProposal,
       endProposal, vote, getProposal, getCandidate, getResults, etc.

   Key Functions:
     • fetchProposals() — Reads all proposals from the blockchain, fetches
       candidates for each, checks if the active account has voted.
     • castVote(proposalId, candidateIndex) — Sends a vote transaction to
       the smart contract.
     • switchAccount(index) — Switches between Hardhat test accounts for
       demonstrating multi-user voting.

   Contract Configuration (contractConfig.js):
     • CONTRACT_ADDRESS — The deployed contract address (set after running
       the deploy script).
     • CONTRACT_ABI — Human-readable ABI containing all contract function
       signatures and event definitions.
     • HARDHAT_CHAIN_ID — Chain ID 31337 (0x7a69) for the local network.
     • HARDHAT_NETWORK — Full network config for adding Hardhat to MetaMask.

11. DISCONNECT FLOW
────────────────────────────────────────────────────────────────────────────────

   When the user clicks "Disconnect Wallet":
     • The account, network, and chain ID states are cleared.
     • The localStorage `walletConnected` flag is removed.
     • Any error messages are cleared.
     • The UI reverts to the disconnected state with the "Connect" button.

   Note: This is an application-level disconnect. MetaMask itself does not
   have a programmatic disconnect API — the site permission remains in
   MetaMask until the user manually revokes it from MetaMask settings.

13. SECURITY CONSIDERATIONS
────────────────────────────────────────────────────────────────────────────────

   • Nonce-based signing prevents replay attacks — each sign request has a
     unique nonce and timestamp.
   • The backend validates that the recovered signer address matches the
     claimed address, ensuring wallet ownership proof.
   • Nonces are invalidated after single use, preventing reuse of old
     signatures.
   • The auto-reconnect mechanism specifically checks for the MetaMask flag
     to avoid silently connecting to Hardhat's injected provider.
   • No private keys are ever handled by the application — all signing is
     done within MetaMask.

15. CONCLUSION
────────────────────────────────────────────────────────────────────────────────

   MetaMask has been successfully integrated into the DGS project with a
   complete wallet connection, authentication, and smart-contract interaction
   pipeline. The useWallet custom hook handles all MetaMask communication
   including detection, connection, network management, message signing, and
   real-time event listening. The WalletConnect page provides a polished
   user experience with clear visual feedback at each step. The backend
   reference server demonstrates how to verify wallet ownership on the
   server side using cryptographic signature verification. The integration
   follows Web3 best practices and is designed to be extensible for future
   wallet providers and blockchain networks.
